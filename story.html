Founder's letter

The making of Synapse

Every decision, every wrong turn · synapse · synapse-service · synapse-web · 2026
Waitlist ⚗ The Lab

What follows is an honest account of every decision, every wrong turn, and every moment where the plan changed. Not a polished case study. Not a launch post. The real story — written while it's still fresh, because the next version of this won't remember the confusion the first one required.

Origin

Why it started

I read books, watch films, listen to podcasts, take notes — and none of it connects. Every highlight lives in isolation. Every idea fades alone in a notebook that never opens again. I kept noticing it: something I read six months ago would suddenly relate perfectly to something I was thinking about today, but I had no way to see that. The connection existed — I just couldn't find it.

I wanted a tool that would show me when two things I had absorbed at different times were talking to each other. A tool that would surface the invisible structure of my own thinking. That tool didn't exist the way I needed it to exist. So I built it.

Synapse is three things: a local-first knowledge graph (synapse — the Flutter app), a cloud sync and AI backend (synapse-service), and a public presence (synapse-web). They were built in parallel, each with its own set of decisions and problems.

⬡ Architecture
Decision 1

Local-first, always

Before writing a single line of code, I made the most important architectural decision: the graph lives on the device, not on a server. This sounds obvious. It has consequences everywhere.

It means Hive, not SQLite. Hive is a pure-Dart key-value store — no native compilation overhead, no schema migrations, just typed Dart objects stored as bytes. Graph traversal is a simple key lookup. I tested SQLite first. It worked, but every query felt like fighting the wrong model. I was shoe-horning a non-relational graph into relational tables. Every new node type would need a migration. Hive let the graph evolve freely.

Local-first also means the backend is never the source of truth. The backend is a backup layer and an AI endpoint — nothing more. No internet, no problem. Synapse works completely, offline, always. This protected the app's soul from day one: it can never become one of those apps that shows you a spinner when the server is down.

Hive over SQLite device = source of truth backend = backup + AI only
Decision 2

Flutter — because the graph needs a canvas

The core UI of Synapse is a force-directed knowledge graph. Nodes repel each other, edges pull connected ones together, the whole thing moves and breathes in response to gestures. This is a custom canvas problem. It rules out anything with a bridge.

React Native's JavaScript-to-native bridge would add latency to every canvas frame. SwiftUI would lock the app to iOS forever. Flutter owns the canvas entirely — 60fps, no bridge overhead, one codebase for iOS and Android. Dart's strong typing means the graph data model is safe to evolve without runtime surprises. The decision took about ten minutes. I haven't second-guessed it once.

"Flutter doesn't render through UIKit. It draws directly. For a graph that needs to move at 60fps, that's not a feature — it's a requirement."

⬡ Backend
Decision 3 · The pivot

Dropping Firestore, keeping Firebase Auth

The original plan: Firebase + Firestore + Cloud Run. Full Google stack, everything connected, no infrastructure to manage. I got about 60% through implementation before I looked at the pricing calculator.

A knowledge graph with 500 nodes syncing changes in real time generates thousands of Firestore read/write operations per session, per user. The math didn't work. Firestore charges per operation at the document level — it's designed for simple key-value access patterns, not graph structures with many small writes. I killed it.

I kept Firebase Auth. This might seem inconsistent — drop the database but keep the auth? It isn't. Auth is the hardest thing to build correctly: token refresh, session expiry, revocation, edge cases under flaky networks. Firebase handles all of it for free on the Spark plan. The Flutter app gets an ID token, sends it as a Bearer header, and the backend verifies it with Firebase Admin SDK. Zero custom auth code. The trade-off is a Google dependency in the identity layer. That's a dependency I'm genuinely happy to accept.

Firestore → dropped Firebase Auth → kept cost predictability over convenience
Decision 4

Fastify + PostgreSQL on Railway

Without Firestore, I needed a backend I controlled. I chose Fastify — the fastest Node.js HTTP framework, TypeScript, schema-driven validation — and PostgreSQL on Railway for the database.

The key insight was that I don't need a relational schema for user data. I just need to store the graph as a blob. PostgreSQL's JSONB type means I can push the entire graph as a single payload and pull it back. The app owns the shape; Postgres owns the durability. When the graph model evolves, only the app changes — the server stores bytes and returns them.

The schema: two tables. users with Firebase UID as primary key, and snapshots in JSONB, keeping the last 10 per user. No migrations. No relational graph tables. Railway gives a predictable flat monthly cost — the exact opposite of Firestore's per-operation pricing.

The API surface: 5 routes. Health check, user sync on first login, push snapshot, pull latest snapshot, and the AI suggest endpoint. That's the entire backend.

Fastify over Express JSONB snapshots Railway over Cloud Run last 10 snapshots per user
⬡ AI Layer
Decision 5 · The model switch

From Claude to Gemini

The first implementation used Claude claude-opus-4-5. The suggestions were good — specific, cross-domain, grounded in the actual content of notes. But when I looked at the stack holistically, the answer changed.

I'm already using Firebase Auth, which is Google. I'm already in the Google ecosystem. The AI layer that fits the architecture is Gemini. Same billing, same cloud ecosystem, reduced vendor fragmentation. Gemini's reasoning quality on cross-domain connection tasks is excellent. The "Because:" explanation — the key feature of AI suggestions — requires a model that understands semantic relationships, not just keyword proximity. Gemini delivers this.

The switch wasn't about model quality. It was about coherence. A product built on fewer moving parts from fewer vendors is easier to maintain, debug, and scale. Google AI alongside Firebase Auth is a natural fit. The architecture diagram in the Lab now says Google AI API — not as a sponsor, but as a deliberate choice.

Gemini over Claude ecosystem coherence on-demand only, not automatic
Decision 6 · UX

The AI suggestions tab

The suggestions feature needed its own space. I added a 4th tab to the bottom nav with a custom Synapse logo icon drawn entirely in Flutter's CustomPainter — a hub-spoke graph with the terra-colored center node, matching the brand icon precisely.

The interaction: swipe right to accept, swipe left to dismiss. Each suggestion shows two connected nodes, a "Because:" explanation from the model, and the source of each note. Accepted connections create a real edge in the knowledge graph. Dismissed ones vanish. The key principle: AI never auto-creates connections. Every edge requires a human accept. The graph is always a reflection of your thinking, not the model's.

Four states were designed: empty (animated Synapse logo, pulsing, with a "Find connections" CTA), loading (shimmer skeleton cards with an animated "Analysing…" dot sequence), suggestions (swipeable cards with a reviewed section below), and all-caught-up (a check mark and refresh option). Each state required its own animation and layout — the loading shimmer alone took three iterations to feel right.

swipe to accept/dismiss human always in the loop 4 distinct states
⬡ The Model
Decision 7 · Growing the model

Beyond books — SourceType

The original model had only Book as a source. Notes belonged to books. But Synapse's value was never about books specifically — it's about connecting ideas from anywhere you absorb them.

I added a SourceType enum with five values: book, movie, serie, personal, topic. Each type carries a label, an icon, and a context-aware creator label — Author for books, Director for films, Creator for series, About for personal notes, Field for topics. The form field hint text adapts. The type picker appears at the top of the editor.

The Book model gained a type field. The fromMap method defaults to SourceType.book when the key is missing — existing Hive data migrates silently, no wipe required. The home screen now says "YOUR LIBRARY" instead of "YOUR BOOKS", and "Sources" instead of "Books" in the stat card. Small label changes that signal a different scope.

5 source types backward-compatible Hive migration context-aware field labels
⬡ The Web
synapse-web

The public face

While building the app, I needed a public presence. Two pages, one design system: the glass morphism language of the app ported to the web — backdrop-filter: blur(20px), ambient orb backgrounds, the same color tokens, the same font stack.

index.html is the waitlist. Email capture, a physics simulation background graph with mouse repulsion, social proof, an EN/ES language toggle. Both subscribe forms are temporarily disabled while the backend email integration isn't live. A decision: show real constraints honestly rather than fake functionality.

lab.html is the technical story. Architecture decisions, tech stack breakdown, an interactive knowledge graph canvas, an SVG architecture diagram with animated flow lines, the decision log, a Hive data schema diagram, and a roadmap with honest status labels. Transparent on purpose — showing the reasoning behind every choice builds more trust than hiding it.

Deployed on Netlify, auto-deploying from the imIsmaelHdz/synapse-web GitHub repo on every push to main. Zero-config deploys — push and forget.

Netlify auto-deploy glass morphism design system transparent by default
⬡ The Real Part
No build goes cleanly

The bugs and the fixes

Everything above is the clean version. Here's what actually happened.

Dev server pointing to the wrong directory
The launch config used "." as the serve path, which resolved to the Flutter project root — not the web folder. Chrome kept showing Flutter's generated index.html instead of the waitlist page. Three rounds of "it's not updating" before finding the root cause.
Fixed by passing the explicit absolute path to npx serve and killing the stale process with lsof -ti:4321 | xargs kill -9.
allNotes getter doesn't exist
When building the AI suggestions screen, I referenced widget.db.allNotes.length. The getter is called notes. The analyzer caught it at compile time — but only after a failed hot restart.
One-line fix. Changed to widget.db.notes.length.
Unused _dotTimer field warning
In the loading state animation of the suggestions screen, I declared late final _dotTimer = _startDotTimer() as a field but never read the variable. Dart's analyzer flagged it as unused. Not a crash — but a warning that shouldn't exist.
Removed the field and called _tickDots() directly from initState().
Lab nav button invisible on the main page
When first adding the Lab link to the waitlist nav, it didn't appear — the inline <a> styling conflicted with the nav's flex layout. Spent too long checking the wrong things before finding the CSS specificity issue.
Applied the .nav-link class consistently to both pages, then matched the active state with .nav-link.active.
Social proof number in too many places
The "247 readers" number was hardcoded in three separate places: the HTML structure, the T[] i18n array, and the subscribe handler JavaScript. Removing it required hunting all three. Found the last one only after the page still showed the number on language toggle.
Removed all three instances. The social proof section now says "readers already on the waitlist" without any number until it's real.
Gemini references scattered across the Lab page
After switching the AI model from Claude to Gemini, the old model name appeared in six separate places: the hero chip, the tech card title and description, the architecture SVG text, decision #4 question/body/verdict, a roadmap item label, and six entries in the T[] i18n array (English + Spanish). A global find-and-replace would have broken IDs and CSS selectors.
Targeted each instance individually. Updated the tech card logo to a Google-blue G, changed ANTHROPIC API to GOOGLE AI API in the SVG, and rewrote the decision body to reflect the actual reasoning for the switch.
SwiftPM resolved packages showing as untracked
When Flutter resolves iOS Swift package dependencies, Xcode writes resolution state into project-local directories. These showed as untracked in git status and are large, auto-generated, and platform-specific. Not obvious at first whether they should be committed.
Left them out of commits. They're resolved fresh on each machine. The .gitignore handles them once you know to look.
google-services.json and GoogleService-Info.plist sensitivity
Firebase config files contain API keys that are technically public (Firebase security comes from Security Rules, not key secrecy), but committing them to a public repository is still a deliberate decision that deserves thought — not an accident.
Excluded from committed changes for now. Documented in the build notes. They'll be added via environment-specific secrets when the backend is deployed.
Hive data persisting across SourceType migration
After adding the type field to the Book model, existing Hive boxes stored maps without the 'type' key. If fromMap threw on missing keys, every existing user's data would be unreadable on first open after the update.
Made fromMap default to SourceType.book when the key is absent. Silent, backward-compatible migration — zero data loss.
⬡ What's next
Status — 2026

Where it stands

The app is feature-complete for private beta. The knowledge graph works — create nodes, write notes, draw wikilinks, watch edges appear. Firebase Auth works. Cloud sync pushes and pulls JSONB snapshots. The AI suggestions tab has its full UI and interaction model — wired to mock data while the backend endpoint is being deployed. The web presence is live at thesynapsetool.netlify.app.

What's still left before public launch:

Deploy synapse-service to Railway. The backend is fully built, tested locally, and ready — it just hasn't been pushed to a production environment. The Railway project exists. This is a configuration step, not a code step.

Connect the domain. thesynapsetool.com still needs to be pointed at Netlify's DNS. Another configuration step.

Wire the AI endpoint. The suggestions tab sends mock data. The backend route exists. Connecting them is a one-file change in the Flutter app — swapping the simulated delay for a real API call to POST /v1/ai/suggest.

Android. Flutter makes this structurally simple — same codebase, different build target. The platform work for Firebase is already done. This is the last item before calling it cross-platform.

"The graph grows. The connections multiply. That was always the point — not the app, but the thinking it makes visible."
